# 1 Github Flow
## Git Branch 전략이란?
Git Branch 전략은 깃 브랜치에 규칙을 부여한는 것.
어떤 Branch를 어디에서 생성하고 병합할지 규칙을 정해놓고 해당 규칙을 팀원들이 지켜가며 개발을 진행하는 것.

## 왜 필요할까?
프로젝트의 효율적인 진행과 관리를 위해
모든 팀원이 각 브랜치가 어떤 역할을 하는지 공통의 이해를 가지게 된다.

## 우리 팀이 정한 전략은, Github Flow
Github Flow에는 2개의 브랜치 존재
- main 브랜치에만 엄격한 규칙 적용
- main을 제외한 나머지 하나의 브랜치는 자유롭게 생성
- main: 언제든 배포가 가능한 상태를 유지

우리 팀은 web 개발을 하고 있기에 다양한 버전 고나리가 필요한 모바일 앱 개발에 적합한 Git Flow보다 Github Flow 선택!

# 2 WebRTC
## Web Real-Time Communication
: 별다른 소프트웨어 없이 카메라, 마이크 등을 사용하여 실시간 커뮤니케이션을 제공해주는 기술.

## peer-to-peer 통신
하나의 컴퓨터와 하나의 컴퓨터가 데이터를 주고 받는 형식.
동등 계층 간 클라이언트/서버의 개념 없이 동등한 노드들로 구성되어 데이터를 주고 받는다.

## 왜 WebRTC인가?
서버를 통하지 않고 직접 통신할 수 있어 서버 과부화 및 비용 문제 해결 가능!

## Singnaling server
유저 간 연결을 하기 위해 요구되는 정보들을 교환하도록 도와준다.
시그널링을 할 때 WebSocket 또는 XMLHttpRequest등을 사용할 수 있다.
SDP: A, B 간의 음성, 영상, 비디오 등 스트리밍 규약을 맞추는 과정
ICE: A, B가 서로 통신할 수 있는 최적의 경로를 찾을 수 있도록 도와주는 프레임워크.

## STUN server
ICE에 대한 정보를 주는 server.

## TURN server
백업 서버. peer to peer가 실패했을 때 사용할 수 있는 서버.

## WebRTC API
- getUserMedia: 사용자 단말기의 미디어 장치(마이크와 웹캠)를 액세스할 수 있는 방법을 제공한다. getUserMedia를 통해 미디어 장치를 액세스 하게 되면 미디어 스트림 객체를 얻을 수 있으며 이를 PeerConnection에 전달하여 미디어 스트림을 전송하게 된다.

- PeerConnection: 가장 중요한 API 이면서 Peer 간의 화상과 음성 등을 교환하기 위한 거의 모든 작업을 수행하는 API 이다. 기본적인 기능은 Singal Processing, Security, 비디오 encode/decode, 네트워크와 관련된 NAT Traversal, Packet send/receive, bandwidth estimation 등이 있다.

- DataChannel: Peer 간에 텍스트나 파일을 주고 받을 수 있는 메시징 API 이다. 설정에 따라 SCTP 또는 RTP 로 전송할 수 있다. DataChannel 은 WebSocket 과 같은 수준의 API 를 제공하며 이는 Row Level API 라 할 수 있다. 대용량 파일을 주고받기 위해서는 이 API 를 활용한 어플리케이션 단의 테크닉이 필요하다.

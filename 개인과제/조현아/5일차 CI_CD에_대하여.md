## CI/CD에 대하여
CI/CD는 지속적인 통합과 지속적인 제공/배포를 의미하여, 소프트웨어 개발 및 배포 과정을 자동화하여 효율성을 높이는 방법론이다.

### CI(Continuous Integration)
여러 개발자가 작성한 코드를 변경 사항에 빈번하게 통합하고, 통합된 코드에 대한 자동화된 테스트를 수행하여 충돌을 방지하고 품질을 유지하는 프로세스이다. 
쉽게 말하면, 애플리케이션의 버그 수정이나 새로운 코드 변경 사항이 주기적으로 빌드 및 테스트되면서 공유되는 레파지토리에 통합(merge) 되는 것이다. 

### CD(Continuous Delivery/Deployment)
지속적인 통합을 통해 준비된 코드를 자동으로 배포하거나, 배포 준비를 자동화하여 수동 개입 없이 안정하게 프로덕션 환경에 배포하는 것을 목표로 하다.
쉽게 말하면, CI에서 build와 test가 완료된 후, 배포 단계에서 릴리스 준비가 되었는지, 그리고 문제가 없는지를 검토한다.
이 과정을 통해 나온 결론이 "이제 사용자들에게 서비스를 제공해도 되겠다!"라는 결론이 나오면 수동적으로 배포를 진행하게 되는데 이것이 CD(Continuous Delivery) 지속적인 제공이다.
반면에, 준비가 완료되자마자 자동화를 통하여 곧바로 배포를 진행하는 경우를 Continous Deployment(지속적인 배포)이다.   

## CI/CD의 목적
### 개발 생산성 향상
개발자는 코드 변경 사항을 빈번하게 통합하고 테스트하며, 배포 준비를 자동화함으로써 개발 속도를 높일 수 있다. 

### 품질 향상
자동화된 테스트를 통해 버그를 조기에 발견하고 수정하여 소프트웨어 품질을 향상시킬 수 있다.

### 배포 속도 단축
배포 과정을 자동화하여 배포 시간을 단축하고, 신속하게 새로운 기능을 사용자에게 제공할 수 있다.

### 비용 절감
자동화된 프로세스를 통해 인적 자원의 투입을 줄이고, 배포 관련 비용을 절감할 수 있다.

## CI 방법
### 코드 변경사항을 주기적으로 merge
작업을 작은 단위로 나눈 후 주기적으로 자주 통합해 코드 충돌을 최소화하는 방법 
1. 개발자들이 GitHub 등에 코드를 통합한다
2. 통합한 코드가 정상적으로 작동하는지 빌드 및 테스트를 진행한다
3. 버그가 발생하면 다음에 처리 할 목록에 정리해두고, 가능한 시점에 해결한다. 

### 통합의 자동화
위의 과정은 귀찮다는 단점이 있다. 따라서. 자동화를 도입하면 GitHub에 코드만 올리면 테스트와 빌드는 프로그램이 자동으로 처리해준다. 
1. 개발자들이 GitHub 등에 코드를 올린다.
2. 빌드 및 테스트는 자동으로 진행되므로, 버그가 생기면 다음날 확인해서 버그를 수정한다. 

### CI의 장점
- 코드의 검증에 소요되는 시간이 줄어든다
- 개발 편의성이 향상된다
- 향상 테스트 코드를 통과한 코드만 레포지토리에 올라가기 때문에, 코드 퀄리티를 높게 유지할 수 있다. 

## CD를 적용한 후의 흐름
1. CI를 적용하여 코드를 검증한다.
2. 실제 배포 환경과 비슷한 환경에서 검증을 진행한다
3. 검증된 소프트웨어를 실제 프로덕션 환경으로 배포한다.
